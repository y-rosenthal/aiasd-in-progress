14  14. Step 3: install the packages (with npm)

The file package.json contains information about which node packages are needed for this project as well as a few other things. The contents of the package.json file for this repo appears below. We will discuss the contents below. We are interested no in the sections “dependencies” and “devDependencies”.

Each line in those sections highlight a specific Node.js package that needs to be installed. The “dependencies” section is for packages that are necessary for running the working project. The “devDependencies” section is for packages that are necessary for developing (i.e. coding) the project but are not needed for running the working final version of the project.

To install these packages, run the following command:

npm install

This command creates a new folder named “node_modules” and downloads the code for each package (AKA module) into that folder.

The package.json files contains version numbers for the packages you are installing (see more about this below).

When a package-lock.json file is present, npm install resolves to the locked versions; the specifiers described below control resolution when updating/adding and how ranges are interpreted.

Note: You may see warnings when you run this command saying that some of the packages that you are trying to install have been deprecated. This is because the packages in the package-lock.json file are out of date.

15 Overview

package.json accepts many kinds of version specifiers. Below are the common forms with brief examples.

15.1 1) Exact version

  • 1.2.3
{ "dependencies": { "lodash": "1.2.3" } }

15.2 2) Caret range (^) — allow non-breaking updates (same major)

  • ^1.2.3 → >=1.2.3 <2.0.0
  • ^0.2.3 → >=0.2.3 <0.3.0 (major=0 is special)
{ "dependencies": { "lodash": "^4.17.21" } }

15.3 3) Tilde range (~) — allow patch updates (same minor)

  • ~1.2.3 → >=1.2.3 <1.3.0
{ "dependencies": { "lodash": "~4.17.0" } }

15.4 4) X-range / wildcard

  • 1.2.x, 1.x, *
{ "dependencies": { "lodash": "1.2.x" } }

15.5 5) Hyphen range

  • 1.2.3 - 1.4.0 → >=1.2.3 <=1.4.0
{ "dependencies": { "lodash": "4.17.0 - 4.17.21" } }

15.6 6) Comparator sets

  • Use >, >=, <, <= and combine with spaces (AND) or || (OR):

    • >=1.2.3 <2.0.0
    • ^1.2 || ^2.0
{ "dependencies": { "lodash": ">=4.17.0 <5" } }

15.7 7) Pre-releases

  • Append identifiers like -alpha.1, -beta, etc.
  • Must match explicitly (e.g., ^2.0.0-beta.1 won’t include 2.0.0 unless allowed by the range).
{ "dependencies": { "react": "^19.0.0-rc.0" } }

15.8 8) Tags (dist-tags)

  • Named pointers published to the registry: latest, next, beta, etc.
{ "dependencies": { "typescript": "next" } }

15.9 9) Aliases (npm:)

  • Install one package as another name.
{ "dependencies": { "lodash": "npm:lodash-es@^4.17.21" } }

15.10 10) Local paths

  • file: to a directory or tarball on disk.
{ "dependencies": { "my-lib": "file:../my-lib" } }

15.11 11) Workspace references (workspace:)

  • For npm workspaces; pin to a workspace package.
{ "dependencies": { "ui": "workspace:*" } }

15.12 12) Local links (link:)

  • Symlink to a local folder (similar to npm link).
{ "dependencies": { "tooling": "link:../tooling" } }

15.13 13) Tarball URLs

  • Remote .tgz URL.
{ "dependencies": { "mypkg": "https://example.com/mypkg-1.2.3.tgz" } }

15.14 14) Git references

  • Full or shorthand; may include #branch, #tag, or #commit.
{
  "dependencies": {
    "lib-ssh": "git+ssh://git@github.com/user/repo.git#main",
    "lib-https": "git+https://github.com/user/repo.git#v1.2.3",
    "gh-short": "github:user/repo#abc1234",
    "gl-short": "gitlab:user/repo#v1.0.0",
    "bb-short": "bitbucket:user/repo#develop"
  }
}

15.15 Quick reference table

Form Example Meaning
