17  14. Step 2: Installing the Node.js packages with npm

There are many “packages” of JavaScript code that have been developed by organizations or individual developers which are then shared with the Node.js community. These packages offer developers of new software to get a running start and avoid needing to write generally useful code from scratch.

Node.js includes a tool, npm. Which stands for the “Node Package Manager” tool. This tool is used to install and manage these packages (as we’ll see below). Therefore technically these packages are known as npm packages (and not Node.js packages).

The code for all npm packages can be found online at https://www.npmjs.com/. To see the Javascript code for a package. Search for the package, then click on the name of the package and click the “code” button.

17.1 How to use a npm package in my Node.js project?

To use a package in your project, you must do both of the following:

  • List the name of the package in the package.json file.
    This file should be in the top folder (i.e. “root folder”) for the project.
    (See more about the exact format of the package.json file below).

  • Run the npm install command in the terminal from the root folder for the project.
    The npm install command downloads the packages that are used by this project into a folder named node_packages. (We’ll discuss this command more below).

17.2 More about the package.json file

Every project that uses Node.js and npm

  • should contain a file named package.json in the top (i.e. root) folder for the project

  • might also contain a file named package-lock.json

17.3 run npm install to in stall the packages

To install these packages, run the following command from the top level folder (i.e. the “root” folder) in the project. This command creates a new folder named “node_modules” and downloads the code for each package into that folder.

npm install

Note: You may see warnings when you run this command saying that some of the packages that you are trying to install have been deprecated. This is not a problem for us now. This usually happens because the packages listed in the package-lock.json files (which we discuss below) file are out of date. Keep reading …

17.4 npm package version numbers

Every time an npm package is updated by the developers of the package a new “version number” is created for that new version of the package. When someone uses a package to write their software, they should record the version number of of the package that was used. This is important because sometimes later versions of the package might not work with the software that was written to use an older version of the package.

Package updates (and hence version numbers) fall into one of the following categories:

  • MAJOR updates: new features that are not necessarily totally compatible with existing software that used older versions of this package.

  • MINOR updates: new features that are intended to be backward-compatible with older versions

  • PATCH fixes: ie. bug fixes that are intended to be backward-compatible with older versions

The npm package version numbers reflect the type of changes that were made to the package since the last version. THe version numbers in the form: MAJOR_NUMBER.MINOR_NUMBER.PATCH_NUMBER
For example the version number 0.7.1 has

  • MAJOR_NUMBER 0
  • MINOR_NUMBER 7
  • PATCH_NUMBER 1

When a new version of a package is released, the major, minor and patch numbers in the version number are updated accordingly.

17.5 using a “^” before a version number (e.g. “^1.1.2”)

It is common for version numbers to be listed with “special characters” that direct npm to install possibly later versions of the package than the specified version.

For example, a caret, i.e. ^, before a version number, e.g. ^1.1.2 , directs npm to install “this version or a later version that has the same MAJOR number”. This is often desired since versions of a package that have the same MAJOR version number should be compatible with each other.

For example the following line:

    "@radix-ui/react-checkbox": "^1.1.2",`

indicates that this project requires version 1.1.2 or later of the package. However, this will only install the latest 1.X.X version. It will NOT install any version from 2.0.0 or later.

The logic for this is that the original developer of our project probably developed the code for the project using version 1.1.2 of the package. If in the future a new improved version 1.1.3 would be released to fix some bugs in the package then that version should be used instead of 1.1.2. The same would be true if even later versions of the package would be release such as 1.1.4 or 1.1.5 or even 1.2.0 - the latest of any 1.X.X version would be used.

However, if a new MAJOR version is released, e.g. 2.0.0 then that would NOT be used when doing npm install for this project. The reason is that a new MAJOR version is not guaranteed to work with code that was written for a prior MAJOR version number of the package.

17.6 package-lock.json records the exact version numbers that were used

As mentioned earlier, in theory, PATCH and MINOR updates to a package shouldn’t break code built and tested with an earlier version that shares the same MAJOR number. In practice, though, that’s not always true. To minimize potential issues, it’s best to install the project using the exact same package versions as those used by the original developer. Doing so ensures you’re working in the same environment and helps you avoid unexpected differences in behavior.

For this reason, when npm installs a package into the project, it records the EXACT version number that was installed into a file named package-lock.json. If the package-lock.json file exists then npm will install whatever version is registered in the package-lock.json file. However, if there is no package-lock.json file in a project then the rules stated above regarding which package is installed are followed. To be clear:

npm install will do the following:

  • If a package-lock.json file exists then the versions recorded in that file are downloaded to the node_modules folder.

  • If there is no package-lock.json file in the project then the versions of the packages as indicated in the package.json file are downloaded to the node_modules folder. Note that these could be later versions than the actual numbers listed in the file. For example if the package.json file lists version ^1.7.3 then the latest version number that still has a 1 in the first position will be installed. (see above for more info about version numbers in the package.json file)

17.7 Depricated warning messages when doing npm install

You may have noticed above that when you ran npm install, you got warning messages saying that one or more packages have been “depricated”. This means that there may be PATCH releases that fix bugs in the version of the package that you installed. This could happen if the package-lock.json file that was in the git repo has versions of packages that had later releases of new versions after the package-lock.json was created.

This generally should not be a problem for a learning tutorial. However, if you are planning on using this code in “production” it is recommended that you delete the package-lock.json file and re-run npm install. This should install the later versions of those packages that may have fixed some bugs. Hoewver, be aware the the original developer did not use these new package versions and therefore the code you have might not work 100% as expected and you might need to spend a little time debugging and testing the application.

17.8 Additional rules for package version numbers in package.json file

Above we discussed version numbers that start with a ‘^’. There are other types of “special characters” that affect which version of a package is actually installed. Below are the common forms with brief examples.

The following is a quick reference table. These are explained in more detail below.

Form Example Meaning
Exact 1.2.3 Only that version
Caret ^1.2.3 Non-breaking updates (same major)
Tilde ~1.2.3 Patch updates (same minor)
X-range 1.2.x, * Wildcards
Hyphen 1.2.3 - 1.4.0 Inclusive range
Comparators >=1.2.3 <2 Logical AND/OR with | |
Pre-release ^2.0.0-rc.1 Includes matching prereleases
Tag "latest", "next" Registry dist-tag
Alias npm:pkg@^1 Install another package under this name
Local file file:../dir From local disk
Workspace workspace:* Workspace package reference
Link link:../dir Symlink to local dir
Tarball URL https://…/pkg.tgz Install from remote tarball
Git github:user/repo#tag From VCS (branch/tag/commit)

17.8.1 1) Exact version number (no extra special characters)

  • 1.2.3
{ "dependencies": { "lodash": "1.2.3" } }

17.8.2 2) Caret range (^) — allow non-breaking updates (same major) - discussed above

  • ^1.2.3 → >=1.2.3 <2.0.0
  • ^0.2.3 → >=0.2.3 <0.3.0 (major=0 is special)
{ "dependencies": { "lodash": "^4.17.21" } }

17.8.3 3) Tilde range (~) — allow patch updates (same minor)

  • ~1.2.3 → >=1.2.3 <1.3.0
{ "dependencies": { "lodash": "~4.17.0" } }

17.8.4 4) X-range / wildcard

  • 1.2.x, 1.x, *
{ "dependencies": { "lodash": "1.2.x" } }

17.8.5 5) Hyphen range

  • 1.2.3 - 1.4.0 → >=1.2.3 <=1.4.0
{ "dependencies": { "lodash": "4.17.0 - 4.17.21" } }

17.8.6 6) Comparator sets

  • Use >, >=, <, <= and combine with spaces (AND) or || (OR):

    • >=1.2.3 <2.0.0
    • ^1.2 || ^2.0
{ "dependencies": { "lodash": ">=4.17.0 <5" } }

17.8.7 7) Pre-releases

  • Append identifiers like -alpha.1, -beta, etc.
  • Must match explicitly (e.g., ^2.0.0-beta.1 won’t include 2.0.0 unless allowed by the range).
{ "dependencies": { "react": "^19.0.0-rc.0" } }

17.8.8 8) Tags (dist-tags)

  • Named pointers published to the registry: latest, next, beta, etc.
{ "dependencies": { "typescript": "next" } }

17.8.9 9) Aliases (npm:)

  • Install one package as another name.
{ "dependencies": { "lodash": "npm:lodash-es@^4.17.21" } }

17.8.10 10) Local paths

  • file: to a directory or tarball on disk.
{ "dependencies": { "my-lib": "file:../my-lib" } }

17.8.11 11) Workspace references (workspace:)

  • For npm workspaces; pin to a workspace package.
{ "dependencies": { "ui": "workspace:*" } }

17.8.12 12) Local links (link:)

  • Symlink to a local folder (similar to npm link).
{ "dependencies": { "tooling": "link:../tooling" } }

17.8.13 13) Tarball URLs

  • Remote .tgz URL.
{ "dependencies": { "mypkg": "https://example.com/mypkg-1.2.3.tgz" } }

17.8.14 14) Git references

  • Full or shorthand; may include #branch, #tag, or #commit.
{
  "dependencies": {
    "lib-ssh": "git+ssh://git@github.com/user/repo.git#main",
    "lib-https": "git+https://github.com/user/repo.git#v1.2.3",
    "gh-short": "github:user/repo#abc1234",
    "gl-short": "gitlab:user/repo#v1.0.0",
    "bb-short": "bitbucket:user/repo#develop"
  }
}

17.9 Alternatives to npm

The npm is the tool used to install and manage these packages. However there are other competing “package manager” tools for “npm packages”. Currently available package manager tools are listed below. There are pros/cons to using each one:

  • npm – default with Node.js; biggest ecosystem. Slower than others.
  • Yarn – fast installs; good caching. Old v1 now legacy.
  • Yarn Berry – no node_modules; great for managing many packages together. Complex setup.
  • pnpm – space-efficient; strict resolution. Some tools misbehave.
  • Bun – ultra-fast; includes runtime. New and less stable.

If you’re just starting out, I recommend using npm (it’s most popular). However, don’t be surprised if some other tutorials that you see mention other package managers. All package managers for Javascript have similar capabilities and features.

17.10 Alternatives to Node.js

Similarly there are Javascript “runtime” environments other than Node.js. Again - if you’re just starting out, I recommend that you use Node.js but be aware of the existance of the others.

  • Deno – modern and secure; runs JavaScript and TypeScript easily. Not all Node tools work yet.
  • Bun – very fast and all-in-one (includes tools for testing and building). Still quite new.
  • Cloudflare Workers – runs small apps close to users for speed. Can’t use all Node features.
  • Workerd – the open-source engine behind Cloudflare’s system. Mainly for advanced use.
